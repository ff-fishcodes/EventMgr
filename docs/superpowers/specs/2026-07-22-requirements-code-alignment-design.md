# 需求文档按当前代码对齐设计

## 1. 背景与确认结论

2026-07-22，用户确认需求文档以当前代码为准，并要求更新所有需求规格相关文件。随后确认采用分层修订方式：当前生效文档完整对齐代码；历史需求文档保留原始正文并增加失效声明、差异摘要和当前基线链接；原始需求与讨论记录保持可追溯性。

本次工作只调整文档，不修改软件行为。

## 2. 目标

1. 消除读者把历史需求误认为当前有效需求的风险。
2. 使当前需求规格准确描述仓库代码已经实现的接口、行为和限制。
3. 统一所有需求文档对 `EventId` 当前规则的说明。
4. 保留原始需求、历史决策及版本演进记录，满足后续回溯要求。
5. 为后续设计、实现和测试提供唯一、明确的当前需求入口。

| 文档类别 | 文件 | 处理策略 |
|---|---|---|
| 当前生效需求基线 | `docs/superpowers/specs/2026-07-21-software-requirements-baseline.md` | 逐项复核代码并修订；作为唯一当前需求基线 |
| 最近一版历史需规 | `docs/superpowers/specs/2026-07-06-software-requirements-spec.md` | 保留正文；在文首标记已被当前基线取代，并列出关键差异 |
| 更早历史需规 | `docs/superpowers/specs/2026-06-26-software-requirements-spec.md` | 保留正文；在文首标记已被当前基线取代，并列出关键差异 |

## 4. 当前事实来源

需求结论按以下优先级核对：

1. 当前工作树中的后端和前端源码；
2. 工程构建文件及可重复执行的构建命令；
3. 已通过源码追踪和构建验证的当前需求基线、概要设计和详细设计；
4. 历史需求、历史设计和讨论记录，仅用于解释演进过程。

当代码注释与可执行逻辑不一致时，以实际执行路径为准，并把注释差异记录为已知问题，不将注释内容写成已实现需求。

## 5. EventId 当前契约

`EventId` 是 `std::string` 类型别名。当前代码实际生成以下三类标识：

| 事件类型 | 当前格式 | 示例 | 代码依据 |
|---|---|---|---|
| 设备事件 | `deviceName-frameID-alarmField` | `锅炉-3-temp_high` | `ExternalAPI::createAlarm()`、`EventManager::makeEventId()` |
| 关联设备的系统事件 | `deviceName-0-eventName` | `锅炉-0-comm_lost` | 三参数 `ExternalAPI::triggerSystemEvent()` |
| 纯系统事件 | `eventName` | `disk_full` | 两参数 `ExternalAPI::triggerSystemEvent()` |

配套限制也属于当前需求：

- 拼接字段没有对 `-` 进行转义。
- 完整 ID 清除路径按前两个 `-` 解析三段字段。
- 帧号解析使用 `std::atoi()`，非法文本可能被解释为 `0`。
- `backend/event_types.h` 中纯系统事件三段式示例是与实现不一致的旧注释，不构成当前契约。

## 6. 当前需规复核范围

除 EventId 外，当前需求基线还需再次对照代码检查以下内容：

- 设备事件和系统事件的产生、查重、清除顺序；
- 原始等级、有效等级、降级和联动之间的实际关系；
- 屏蔽对存储、联动、日志和通知的影响；
- 动作注册、事件级动作、等级默认动作和禁用键规则；
- 前后端实际部署方式、桥接方式及 Socket 桩状态；
- Qt、C++ 标准、线程池和构建方式；
- 内存存储、日志桩、生命周期、并发及默认 UI 死锁等已知限制；
- 当前代码无法证明的目标平台、性能和持久化能力不得写成已验收能力。

## 7. 导航与版本状态

文档入口应清楚区分：

- “当前有效”：只能指向当前需求基线、当前概要设计、当前详细设计和当前架构图；
- “历史归档”：保留日期、原版本号和原状态，但增加“已被取代”标识；
- “原始输入”：仅表示需求来源，不表示当前实现或当前验收标准；
- “讨论记录”：保存问题、选择、确认、差异和验证结论。

所有历史文档的失效声明统一链接到当前需求基线，避免形成多个互相竞争的有效版本。

## 8. 验证方法

完成修订后执行以下检查：

1. 搜索所有需求相关文件中的 `protocolID`、旧系统事件冒号格式、纯系统三段式格式和旧 Qt 版本，确认它们只出现在明确标注的历史内容或差异说明中。
2. 搜索所有当前有效文档中的 EventId 示例，确认三类格式一致。
3. 检查 Markdown 相对链接和显式锚点，确保导航可达且无重复锚点。
4. 对照 `external_api.cpp`、`event_manager.cpp`、`event_types.h`、`linkage_engine.cpp`、`config_manager.cpp` 和 `frontend.pro` 抽查需求结论。
5. 检查 Git 差异，确认未修改源代码，也未无痕改写历史需求正文。
