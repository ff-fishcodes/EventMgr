## 1. 范围

### 1.1 标识

- **系统名称**：事件管理中心
- **文档编号**：EventMgr-RS-001
- **适用项目**：上位机事件管理子系统

### 1.2 概述

事件管理中心接收下位机上报的健康状态信息，同时监测后端系统自身状态（通信断连、磁盘空间等），将异常生成统一格式的告警事件，按预定义规则执行联动管控，实时呈现给操作员。

联动动作通过独立线程池异步执行，避免阻塞事件处理管线。线程池大小可配置，默认使用全局线程池。

系统支持两种部署方式：前后端分离（通过消息通信）和前后端一体（同进程内桥接）。两种方式切换不修改后端核心代码。

运行平台：飞腾 FT/2000 + 银河麒麟 V10。下位机 5-20 台，状态帧总数 ≤ 100，上报频率 ≥ 1 Hz。

## 3. 需求

### 3.1 能力需求（RX_BASIC_ABILITY）

#### 3.1.1 告警产生与消除（RX_ALARM_LIFECYCLE）

##### 3.1.1.1 功能需求

系统应能够接收来自下位机状态帧的报警信息，或后端自行监测的系统事件，生成带有唯一编号的告警事件，存入活跃事件管理表。支持以下功能点：

- **事件编号**：设备事件编号由"下位机标识-状态帧标识-报警字段"三段组成（如 `1-3-temp_high`）；系统事件编号使用预定义的名称字符串，关联设备时附加冒号和下位机标识（如 `comm_lost:3`）
- **事件来源**：每个事件必须标注来源——是下位机状态帧上报，还是后端自行监测产生
- **事件等级**：分四级——紧急（需立即处理）、严重（需尽快处理）、一般（需关注）、提示（仅供参考）。提示等级不触发联动，但日志和通知照常
- **重复上报抑制**：同一编号的事件在活跃状态下再次上报时，系统必须忽略，不产生新事件、不重复触发联动、不重复通知前端
- **消除不存在的隐患**：消除请求针对不存在的事件时，静默忽略不报错
- **时间戳**：事件产生时自动记录系统本地时间，格式"年-月-日 时:分:秒"，外部调用方不需要提供
- **日志记录**：事件产生和消除时均应写入日志，内容包括设备标识、描述、动作（"发生"或"消除"）
- **事件携带的信息**：事件编号、来源类型、下位机标识、状态帧标识、报警字段名、描述、原始等级、经降级后的实际等级、状态（活跃/已消除）、产生时间

##### 3.1.1.2 输入输出

**输入**：

| 接口 | 说明 |
|------|------|
| 设备告警触发 | 调用方给出下位机标识、状态帧标识、报警字段名、产生或消除标志。系统自动查报警目录获取等级和描述；目录中无匹配时使用默认等级（提示）和字段名兜底 |
| 系统事件触发（纯系统） | 调用方给出预定义事件名、产生或消除标志 |
| 系统事件触发（关联设备） | 与上相同，另加下位机标识 |
| 手动创建事件 | 调用方给出下位机标识、状态帧标识、报警字段、等级、描述，系统返回完整事件对象 |
| 提交事件 | 调用方将事件对象提交到处理管线 |
| 按设备标识消除 | 调用方给出下位机标识、状态帧标识、报警字段 |
| 按事件编号消除 | 调用方给出完整事件编号，系统自动识别是设备格式还是系统格式 |

**输出**：

- 活跃事件表中新增或移除记录
- 产生时：若实际等级非提示，触发联动动作；消除时：触发消除侧联动
- 若未被屏蔽，推送事件变化通知给前端
- 写入日志

##### 3.1.1.3 行为过程描述

事件产生流程：
1. 检查该编号是否已在活跃事件表中存在，存在则忽略
2. 查询是否有降级配置，计算实际生效等级
3. 记录当前系统时间
4. 存入活跃事件表
5. 若实际等级不是提示，执行产生侧联动
6. 联动执行完毕后，若该事件未被屏蔽，推送通知给前端
7. 写日志

事件消除流程：
1. 根据事件编号在活跃事件表中查找，不存在则忽略
2. 标记为已消除
3. 执行消除侧联动
4. 推送消除通知给前端
5. 写消除日志
6. 从活跃事件表中移除

注：屏蔽仅影响前端通知（产生时不推送），联动和日志不受屏蔽影响。重复事件在第 1 步拦截，后续步骤全部跳过。

##### 3.1.1.4 行为参数

| 参数 | 约束 |
|------|------|
| 下位机标识 | ≥ 0 整数，系统事件可为 0 |
| 状态帧标识 | ≥ 0 整数，系统事件为 0 |
| 报警字段名 | 非空字符串 |
| 等级 | 1=紧急, 2=严重, 3=一般, 4=提示 |
| 时间戳 | 由系统自动生成，精确到秒 |
| 查重性能 | 常数时间（约 1 微秒级） |
| 消除不存在的事件 | 静默忽略，不报错 |

#### 3.1.2 事件联动（RX_EVENT_LINKAGE）

##### 3.1.2.1 功能需求

系统应在事件产生和消除时自动执行预配置的联动管控动作。联动配置在系统启动时一次性完成，运行时不再修改。支持以下功能点：

- **能力注册**：每种联动能力有一个名称和对应的执行操作，操作中已封装好所需参数。所有能力在启动时集中注册
- **按事件配置联动**：可为每个事件编号单独指定——产生时执行哪些能力，消除时执行哪些能力。两种配置相互独立，不自动镜像
- **按等级配置默认联动**：可为每个等级设置默认能力列表，该等级所有事件产生时自动附带
- **优先级合并**：事件产生时，实际执行的能力 = 事件单独配置 + 等级默认配置（追加合并）；若两者均为空，使用事件自带的兜底列表。事件消除时，仅使用事件单独配置的消除能力
- **提示等级跳过**：实际等级为"提示"时，不执行任何联动
- **联动桥接**：每次联动执行后，将"事件编号、是产生还是消除"通知给外部。上层将通知转换为界面信号，由使用者按事件编号自行决定锁或解锁哪些界面控件。后端注册层只注册管控类能力（指令发送、蜂鸣器等），不涉及界面操作
- **仅注册后端能力**：注册层只包含向下位机发指令、触发蜂鸣等后端操作，界面锁控由桥接通知在上层另行处理

##### 3.1.2.2 输入输出

| 接口 | 说明 |
|------|------|
| 注册能力 | 给出名称和操作，在启动时调用 |
| 配置事件联动 | 给出事件编号、产生侧能力名称列表、消除侧能力名称列表 |
| 设置等级默认 | 给出等级和默认能力名称列表 |
| 联动桥接通知 | 系统输出事件编号和产生/消除标志，上层使用者自行处理 |

##### 3.1.2.3 行为过程描述

产生侧联动执行：
1. 检查事件实际等级是否为提示，是则直接返回
2. 获取能力名称列表：事件级配置 + 等级默认 → 合并
3. 若合并后仍为空，尝试使用事件自带的兜底列表
4. 按名称逐一查找已注册的能力，提交到联动专用线程池异步执行
5. 末尾调桥接通知（告知哪个事件、正在产生）

消除侧联动执行：
1. 检查事件实际等级是否为提示，是则直接返回
2. 获取事件级配置的消除能力列表
3. 按名称逐一查找，提交到线程池异步执行
4. 末尾调桥接通知（告知哪个事件、正在消除）

联动执行与事件处理管线异步分离：事件处理（查重、降级、存储、通知、日志）在主线程同步完成并立即返回；联动动作在独立线程池中并发执行，不阻塞后续事件的接收和处理。

##### 3.1.2.4 行为参数

| 参数 | 约束 |
|------|------|
| 事件级配置 | 最高优先级，产生和消除独立配置 |
| 等级默认配置 | 与事件级合并，仅对产生侧生效 |
| 实例级兜底 | 最低优先级，仅在前两者为空时使用 |
| 提示等级 | 不执行任何联动，也不调桥接通知 |

#### 3.1.3 告警降级（RX_ALARM_DOWNGRADE）

##### 3.1.3.1 功能需求

操作员可以将某个事件编号的报警等级降低为更低等级。降级按事件编号精确设置，设置后立即生效——后续新事件产生时自动以降级后的等级处理，已存在事件在界面上也立即反映降级状态。

支持以下功能点：

- **设置降级**：指定事件编号和降级后的等级，立即生效
- **取消降级**：恢复该事件的原始等级
- **查询有效等级**：系统根据事件编号和原始等级，返回当前实际应使用的等级
- **影响范围**：降级仅改变等级相关的决策（联动是否执行、前端颜色显示），不影响日志写入和前端通知推送
- **不持久化**：降级配置仅内存有效，软件重启后全部清空，操作员需手动重新设置。这是安全策略——降级是临时豁免，不应跨会话保持

##### 3.1.3.2 输入输出

| 接口 | 说明 |
|------|------|
| 设置降级 | 给出事件编号和降级目标等级 |
| 取消降级 | 给出事件编号 |
| 查询有效等级 | 给出事件编号和原始等级，返回实际生效等级 |

##### 3.1.3.3 行为过程描述

设置降级：
1. 在降级映射中记录：该事件编号 → 降级目标等级
2. 界面下次刷新时自动反映新等级

取消降级：
1. 从降级映射中移除该事件编号的条目

查询有效等级：
1. 查降级映射，存在则返回降级后等级
2. 不存在则返回原始等级

##### 3.1.3.4 行为参数

| 参数 | 约束 |
|------|------|
| 事件编号 | 精确匹配 |
| 降级目标等级 | 1-4 |
| 生效时机 | 设置后立即 |
| 持久化 | 不持久化，重启清空 |

#### 3.1.4 报警屏蔽（RX_ALARM_SHIELD）

##### 3.1.4.1 功能需求

操作员可以设置某个事件编号为屏蔽状态。被屏蔽的事件在产生时不推送前端通知（界面上看不到），但联动动作和日志写入照常执行。

支持以下功能点：

- **设置屏蔽**：指定事件编号，立即生效
- **取消屏蔽**：恢复该事件的前端通知
- **查询屏蔽状态**：判断指定事件是否被屏蔽
- **屏蔽计数**：随时获取当前被屏蔽事件的总数供界面展示
- **影响范围**：屏蔽仅影响前端通知，联动和日志不受影响
- **不持久化**：与降级同理，屏蔽配置仅内存有效，重启后清空

##### 3.1.4.2 输入输出

| 接口 | 说明 |
|------|------|
| 设置屏蔽 | 给出事件编号 |
| 取消屏蔽 | 给出事件编号 |
| 查询屏蔽状态 | 给出事件编号，返回是否被屏蔽 |
| 屏蔽计数 | 返回当前屏蔽数量 |

##### 3.1.4.3 行为过程描述

设置屏蔽：
1. 将该事件编号加入屏蔽集合
2. 后续该事件产生时跳过前端通知步骤

取消屏蔽：
1. 从屏蔽集合中移除该事件编号
2. 后续该事件产生时恢复前端通知

##### 3.1.4.4 行为参数

| 参数 | 约束 |
|------|------|
| 事件编号 | 精确匹配 |
| 影响范围 | 仅前端通知，联动和日志不受影响 |
| 生效时机 | 设置后立即 |
| 持久化 | 不持久化，重启清空 |

#### 3.1.5 系统事件（RX_SYSTEM_EVENT）

##### 3.1.5.1 功能需求

后端自行监测到的事件（通信断连/恢复、磁盘满、CPU过载、关键服务异常等）与下位机上报事件使用同一套管理管线，但事件名必须在预设列表中预定义。

支持以下功能点：

- **预定义列表**：所有系统事件名在一个集中列表中定义，每项含事件名、描述、等级。项目方按需增删
- **名称校验**：触发系统事件时校验名称是否在列表中，不在则警告并忽略
- **纯系统事件**：不与特定下位机关联（如"磁盘满"），编号即事件名
- **关联设备系统事件**：与某下位机关联（如"与下位机3通信断连"），编号为"事件名:下位机标识"，描述自动拼接
- **与设备事件共享管线**：系统事件可参与联动配置、降级屏蔽等，与设备事件无差别

##### 3.1.5.2 输入输出

| 接口 | 说明 |
|------|------|
| 触发纯系统事件 | 给出预定义事件名、产生或消除标志 |
| 触发关联设备系统事件 | 额外给出下位机标识 |

##### 3.1.5.3 行为过程描述

纯系统事件触发：
1. 在预定义列表中按名称查找
2. 未找到则警告并忽略
3. 找到则构造事件，编号=事件名，等级和描述取自预定义
4. 进入通用事件处理管线

关联设备系统事件触发：
1. 同上验证名称
2. 编号 = "事件名:下位机标识"
3. 描述 = "下位机X-事件描述"
4. 进入通用事件处理管线

##### 3.1.5.4 行为参数

| 参数 | 约束 |
|------|------|
| 事件名 | 必须在 system_events.cpp 预定义列表中，否则忽略 |
| 下位机标识 | 仅关联设备时有效，≥ 0 整数 |
| 预定义列表 | 具体内容由项目方自行增删，当前为示例 |

#### 3.1.6 前端展示与配置（RX_FRONTEND_UI）

##### 3.1.6.1 功能需求

前端应提供可视化界面用于实时事件监控和报警配置管理。

**a. 活跃事件列表**
- 表格列：事件编号、描述、产生时间、等级、状态、降级复选框、屏蔽复选框
- 等级颜色：红色=紧急、橙色=严重、黄色=一般、蓝色=提示
- 降级和屏蔽复选框：勾选/取消时配置即时生效
- 底部提供下拉框选择报警定义 + "模拟报警"按钮供开发调试

**b. 报警配置页**
- 表格列：事件编号、描述、原始等级、降级复选框、屏蔽复选框
- 降级/屏蔽复选框勾选后需点"应用"按钮统一提交
- 提供"刷新"按钮重载目录

**c. 实时报警日志**
- 独立控件，两列（时间、报警内容），可嵌入主界面任意位置
- 描述按等级着色

**d. 主容器**
- 含标签页（事件列表 + 报警配置）+ 底部屏蔽数量状态栏
- 可嵌入任意父级窗口
- 切换标签页时自动刷新目标页面

**e. 数据同步机制**
- 后端事件变化时通过桥接层主动推送通知，前端即时刷新（毫秒级延迟）
- 同时保留每秒一次的定时轮询作为兜底
- 两页（事件列表/报警配置）共享同一份降级屏蔽配置，切换时自动同步

**f. 事件通知格式**
后端推送的事件变化通知内容包含：事件类型（产生/消除）、下位机标识、状态帧标识、报警字段名、描述、等级。

##### 3.1.6.2 输入输出

| 接口 | 说明 |
|------|------|
| 获取活跃事件列表 | 返回当前所有活跃事件（含时间戳、等级、屏蔽状态） |
| 获取报警目录 | 返回所有报警定义及其降级/屏蔽状态 |
| 设置降级/取消降级 | 前端操作降级复选框时调用 |
| 设置屏蔽/取消屏蔽 | 前端操作屏蔽复选框时调用 |
| 查询屏蔽数量 | 供状态栏显示 |
| 模拟报警触发 | 供开发调试 |

##### 3.1.6.3 行为过程描述

前端刷新流程：
1. 后端事件变化或定时器到期触发刷新
2. 获取最新活跃事件列表
3. 清空表格，逐行填充数据
4. 降级/屏蔽复选框读取当前配置状态

标签页切换同步：
1. 切换到目标页面时触发
2. 立即刷新目标页面数据

##### 3.1.6.4 行为参数

| 参数 | 约束 |
|------|------|
| 主动推送延迟 | 毫秒级 |
| 定时刷新周期 | 1 秒（兜底） |
| 表格编辑 | 只读（复选框除外） |
| 降级/屏蔽复选框 | 事件列表即时生效，报警配置需点"应用" |
| 所有前端控件 | QWidget 子类，可嵌入任意父级窗口 |

### 3.2 外部接口需求（RX_interface）

#### 3.2.1 接口标识和接口图

```
外部调用方（网络接收模块 / 通信监测 / 状态监测组件）
    │
    ├── 设备告警触发（产生/消除）
    ├── 系统事件触发（产生/消除）
    ├── 手动创建事件
    ├── 提交事件
    ├── 按设备标识消除
    ├── 按事件编号消除
    ├── 查询活跃事件列表
    └── 查询报警目录
            │
            ▼
      事件管理中心
```

#### 3.2.2 对外接口（JK_API）

**接口标识**：`JK_API`

系统对外暴露一个统一的调用入口，提供以下服务：

| 服务 | 说明 |
|------|------|
| 设备告警触发 | 给出下位机标识、状态帧标识、报警字段、产生或消除标志。系统自动查目录获取等级和描述 |
| 系统事件触发 | 给出预定义事件名，系统事件产生或消除 |
| 手动创建事件 | 给出完整参数，返回事件对象 |
| 提交事件 | 将事件对象提交到处理管线 |
| 按设备标识消除 | 给出下位机标识、状态帧标识、报警字段 |
| 按事件编号消除 | 给出事件编号，系统自动解析格式 |
| 查询活跃事件 | 返回当前所有活跃事件 |
| 查询报警目录 | 返回所有报警定义及降级/屏蔽状态 |

- 所有接口同步调用，调用方不等待异步回调
- 无效输入（如系统事件名不在预定义列表、消除不存在的事件）静默忽略，不抛异常

### 3.3 内部接口需求（rx_in_interface）

#### 3.3.1 事件存储管理（RX_IN_EVENTMGR）

负责活跃事件的增删查管理，处理事件产生和消除的完整管线。

| 职责 | 说明 |
|------|------|
| 事件产生管线 | 查重、计算有效等级、记录时间、存储、联动、通知、日志 |
| 事件消除管线 | 查找、标记消除、联动、通知、日志、移除 |
| 查询 | 按编号查找、按设备查找、获取全部活跃事件、活跃事件数量 |

#### 3.3.2 联动引擎（RX_IN_LINKAGE）

负责联动能力的注册管理、事件联动配置、等级默认配置和联动执行。

| 职责 | 说明 |
|------|------|
| 能力注册 | 名称→操作映射，启动时集中注册 |
| 联动配置 | 事件级配置（产生侧+消除侧）、等级默认配置 |
| 联动执行 | 产生侧和消除侧联动执行，末尾触发桥接通知 |

#### 3.3.3 配置管理（RX_IN_CONFIG）

负责降级映射表和屏蔽集合的管理。

| 职责 | 说明 |
|------|------|
| 降级管理 | 设置、取消、查询 |
| 屏蔽管理 | 设置、取消、查询、计数 |

---

### 3.4 内部数据需求（rx_in_data）

- **活跃事件表**：事件编号为键，事件完整信息为值。常数时间查找。仅内存存储，重启后从下位机上报重建
- **降级映射**：事件编号 → 降级后等级
- **屏蔽集合**：被屏蔽的事件编号集合
- **联动能力表**：能力名称 → 执行操作
- **事件联动配置表**：事件编号 → 产生侧和消除侧的能力名称列表
- **等级默认表**：等级 → 默认能力名称列表

所有配置均为内存存储，不持久化。

### 3.5 设计和实现约束（rx_design）

| 编号 | 约束项 | 说明 |
|------|--------|------|
| RX_DESIGN_01 | 运行平台 | 飞腾 FT/2000，银河麒麟 V10 |
| RX_DESIGN_02 | 后端语言/标准 | C++11 + Qt 5.15.2 |
| RX_DESIGN_03 | 前端框架 | Qt 5.15.2 |
| RX_DESIGN_04 | 部署架构 | 支持前后端分离和前后端一体（同进程桥接），架构切换不改后端核心 |
| RX_DESIGN_05 | 事件处理并发 | 事件处理管线为单线程事件循环，无锁设计。联动动作通过独立线程池异步执行，与事件管线并发隔离 |
| RX_DESIGN_06 | 编译标准 | GCC C++11，-Wall -Wextra -Werror |
| RX_DESIGN_07 | 代码规范 | 构造函数/析构函数显式声明，公开接口写注释 |
| RX_DESIGN_08 | 异常输入 | 静默处理，不抛异常 |
| RX_DESIGN_09 | 文档提交 | 提交 git 前完善相关文档 |
| RX_DESIGN_10 | 开发框架 | 后端 C++11 + Qt 5.15.2（QThreadPool/QtConcurrent 用于联动异步执行）；前端 Qt 5.15.2 |
| RX_DESIGN_11 | 内存管理 | 使用裸指针和栈对象，不使用智能指针；静态持有对象在 init() 中创建，进程退出时由操作系统回收 |
| RX_DESIGN_12 | 通知机制 | 后端通过回调注入实现解耦，一体模式通过 Qt 信号推送，分离模式通过 Socket 消息推送 |
| RX_DESIGN_13 | 配置作用域 | 降级和屏蔽配置精确到单个事件编号，不提供全局或按等级批量设置 |
| RX_DESIGN_14 | 前端嵌入方式 | 前端控件为 QWidget 子类，构造函数接受桥接层指针，不依赖全局单例 |
| RX_DESIGN_15 | 线程安全 | 联动回调（Lambda 闭包）可能被多个工作线程并发执行，回调内部涉及共享状态时需自行加锁保护 |

### 3.6 适应性需求（rx_adaptability）

系统应具备以下适应性能力：

| 编号 | 类别 | 需求描述 |
|------|------|---------|
| RX_ADAPT_01 | 下位机数量 | 系统应在 5-20 台下位机规模下正常运行，下位机增减无需修改系统代码 |
| RX_ADAPT_02 | 状态帧扩展 | 新增下位机状态帧或报警字段时，仅需在报警目录中添加定义条目，系统核心逻辑无需修改 |
| RX_ADAPT_03 | 联动能力扩展 | 新增联动管控能力仅需添加一处注册项（名称和回调），联动引擎和执行管线零改动 |
| RX_ADAPT_04 | 前端界面扩展 | 新增前端功能页面可通过向标签页容器添加新页实现，不影响现有页面 |
| RX_ADAPT_05 | 系统事件扩展 | 新增系统事件类型仅需在预定义列表中加一条目，事件处理管线自动支持 |
| RX_ADAPT_06 | 事件等级扩展 | 当前等级体系（1-4级）可扩展为更多层级，新增等级不影响已有等级的定义和行为 |
| RX_ADAPT_07 | 跨平台适应 | 后端代码不依赖特定平台 API，仅使用 C++ 标准库，可移植至其他 Linux 发行版或嵌入式平台 |

### 3.7 安全性需求（rx_security）

系统应在运行过程中保障操作安全，防止误操作和未授权行为：

| 编号 | 类别 | 需求描述 |
|------|------|---------|
| RX_SEC_01 | 降级豁免安全 | 降级是一种对报警等级的临时豁免操作。软件重启后所有降级配置必须自动清除，恢复原始等级。操作员须在每次系统启动后根据当前工况手动重新评估是否需要降级，防止历史降级设置被遗忘而导致严重的报警被长期压制 |
| RX_SEC_02 | 屏蔽豁免安全 | 与降级同理，屏蔽配置在软件重启后必须自动清除。操作员每次启动后手动重新判断是否需要屏蔽，防止历史屏蔽导致关键报警在重启后仍然不可见 |
| RX_SEC_03 | 联动不可绕过 | 屏蔽操作仅影响前端展示，不得影响联动动作的执行。即操作员可以选择"不看某个报警"，但不能选择"不执行某个报警对应的管控动作" |
| RX_SEC_04 | 事件编号不可伪造 | 系统事件名必须在预定义列表中注册，外部调用方不能使用未注册的事件名触发事件。使用未注册名称时系统应警告并忽略请求 |
| RX_SEC_05 | 降级下限约束 | 降级操作只能将等级降低，不得将提示等级提升为更高级别。降级后的等级不得高于原始等级 |
| RX_SEC_06 | 操作可追溯 | 所有事件产生、消除、降级设置、屏蔽设置均须写入日志，以便事后审计追溯操作过程 |
| RX_SEC_07 | 输入保护 | 系统应对所有外部输入进行容错处理。非法参数（负数标识、空字符串、越界等级值）应静默忽略或使用默认值，不得导致系统崩溃或未定义行为 |

---

### 3.8 保密性需求（rx_confidentiality）

系统涉及上下位之间的通信数据及操控指令，需满足以下保密性要求：

| 编号 | 类别 | 需求描述 |
|------|------|---------|
| RX_CONF_01 | 数据隔离 | 事件管理中心作为上位机子系统，其内部数据（活跃事件表、配置表）仅在上位机进程内可见，不暴露给下位机 |
| RX_CONF_02 | 通信安全 | 上下位机之间的指令传输由项目通信模块保障（不在本需求范围），但事件管理中心内部不缓存或记录通信协议中的认证凭据、密钥等敏感信息 |
| RX_CONF_03 | 日志安全 | 日志输出内容应限定为事件描述和处理动作，不得包含通信协议细节、内存地址、栈信息等可能泄露系统内部状态的敏感数据 |
| RX_CONF_04 | 接口隔离 | 系统对外仅暴露 ExternalAPI 定义的接口方法，内部数据结构（如 activeEvents_ 存储容器、降级映射表等）不向外部调用方暴露 |

---

### 3.9 可靠性需求（rx_reliability）

系统应在异常工况下保持稳定运行，不因局部故障导致整体功能丧失：

| 编号 | 类别 | 需求描述 |
|------|------|---------|
| RX_REL_01 | 单设备故障隔离 | 单台下位机通信中断时，该设备的事件正常处理（如产生断联告警），其他下位机的事件不受影响 |
| RX_REL_02 | 异常输入容错 | 重复上报、消除不存在事件、未注册的系统事件名、越界等级值等异常输入应静默处理，不抛异常、不崩溃 |
| RX_REL_03 | 内存稳定性 | 活跃事件表在规模假设范围内（≤100条目），不应出现内存泄漏或无限制增长。事件消除时及时从表中移除 |
| RX_REL_04 | 启动可靠性 | 系统启动时（EventMgrModule::init）一次性完成所有模块实例化和联动注册。若启动过程任一环节失败，应有明确的错误输出便于诊断 |
| RX_REL_05 | 运行时不可变 | 联动配置和动作注册仅在启动时执行一次，运行时不可变更，避免运行中配置变更引入的竞态和不确定性 |
| RX_REL_06 | 通知可靠性 | 前端通知采用主动推送为主、定时轮询为辅的双重机制。推送信号异常丢失时，1 秒定时轮询保证界面最终一致 |
| RX_REL_07 | 联动执行隔离 | 联动动作在独立线程池中异步执行，单个设备的指令发送阻塞（等待 ACK）不影响其他设备的联动执行，也不阻塞事件处理管线 |
| RX_REL_08 | 事件处理确定性 | 事件处理管线（查重、降级、存储、通知、日志）在主线程顺序执行，无竞态条件，行为可复现 |

## 4. 注释

- "下位机"指被上位机监控的物理设备
- "报警"与"事件"可互换使用
- 本文档 RX_ 编号与设计文档中 CSC_/CSU_ 编号构成追踪关系
- 降级和屏蔽配置的"不持久化"设计既属安全性需求也属设计约束，分别在对应章节陈述
